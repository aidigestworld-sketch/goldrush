# Opportunity Engine — Database Schema

## Physical Implementation of GRAPH_SCHEMA_SPEC.md

---

## 0. Assumption Flagged Up Front

**DB choice: PostgreSQL, not a dedicated graph database (Neo4j, etc.).**
This is a real fork, not a detail, so it's stated explicitly rather
than picked silently:

- Your stack is already Fastify/Railway/Postgres-shaped (per prior
  context) — Railway hosts Postgres natively, and introducing a
  second DB engine for one subsystem adds real operational cost for
  a solo founder (backups, migrations, a second connection pool, a
  second thing that can go down).
- The graph's actual query patterns (per GRAPH_SCHEMA_SPEC.md) are
  shallow: traverse one `composes` chain (5 fixed legs), fetch a
  Hypothesis's `evidence_for`/`evidence_against`, walk a short
  `supersedes` chain. None of this needs multi-hop graph traversal
  algorithms (shortest path, centrality, etc.) that would justify
  Neo4j's Cypher engine — recursive CTEs in Postgres handle chains
  this shallow without strain.
- `pgvector` (Postgres extension) covers the one place true
  similarity search is needed — node-matching for resurrection (v1
  §7) and Evidence clustering (GRAPH_SCHEMA_SPEC.md §4.13) — without
  a separate vector DB either.

**If this assumption is wrong** (e.g. you're expecting graph depth to
grow well beyond what's specified, or want Cypher-style queries for
product reasons), say so before continuing — the rest of this
document is built on it.

## 1. Core Design Decision: Table-per-Node-Type, Universal ID Space

Two options exist for mapping 11 heterogeneous node types onto
relational tables: (a) one giant `nodes` table with a JSONB blob per
type, or (b) one physical table per node type with typed columns.

**Decision: (b), table-per-type**, because:

- Node schemas (GRAPH_SCHEMA_SPEC.md §2) are materially different in
  shape (Market has `growth_rate_estimate`, Founder has
  `distribution_assets` — no meaningful shared structure beyond the
  base fields). Forcing them into one JSONB-blob table trades away
  Postgres's actual type checking, indexing, and constraint
  enforcement for a false sense of flexibility.
- **All tables share a universal UUID id space** (not
  per-table auto-increment) specifically so that edges — which must
  reference *any* node type generically (§3 below) — can do so with a
  single `uuid` column instead of a `(table_name, id)` composite key
  everywhere. This is the one deliberate exception to "let Postgres
  types do the work": ids are UUIDs, generated by the app or via
  `gen_random_uuid()`, never serial integers.

## 2. Base Columns (every node table includes these)

```sql
id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
label           text,
created_at      timestamptz NOT NULL DEFAULT now(),
last_seen_at    timestamptz NOT NULL DEFAULT now(),
status          text NOT NULL,        -- CHECK constraint, legal values per table — §5
confidence      real,                  -- nullable where a type has a composite instead (Hypothesis, OpportunityCandidate)
```

`source_refs` is **not** a column on every table. It is represented as
rows in a single generic join table (§4) rather than an array column,
because `source_refs` needs to be queried from the Evidence side too
("which nodes cite this Evidence") — an array column on the citing
side makes that reverse lookup a full table scan; a join table makes
it an indexed lookup either direction.

## 2.5 Pipeline Run (canonical definition lives here; also shown in AGENT_EXECUTION_DAG.md §1)

`opportunity_candidate` (§3.9) has a foreign key into this table, so it
must exist first. Full rationale and the companion
`agent_execution_log` table are in AGENT_EXECUTION_DAG.md §1 — repeated
here only so this document's DDL is self-contained and in FK order.
Note `pipeline_run` itself references `founder` (§3.8), which appears
later in this document's presentation order — actual migration files
should create tables in true dependency order (`founder` →
`pipeline_run` → everything else), not the thematic order used here
for readability.

```sql
CREATE TABLE pipeline_run (
  run_id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  founder_id      uuid NOT NULL REFERENCES founder(id),
  vertical        text NOT NULL,
  current_stage   text NOT NULL DEFAULT 'discovery',
  status          text NOT NULL DEFAULT 'running',
  failure_reason  text,
  started_at      timestamptz NOT NULL DEFAULT now(),
  completed_at    timestamptz
);
ALTER TABLE pipeline_run ADD CONSTRAINT chk_status
  CHECK (status IN ('running','completed','failed','insufficient_evidence'));
```

column and one write at validation time — cheap enough that it's
included here rather than deferred a second time. If you'd rather keep
it out until it's actually needed, say so and it comes out.

## 4. Source References (Evidence citations — generic join table)

Rather than an array column per table, `source_refs` is one polymorphic
join table used by every citing node type except Founder and Outcome
(GRAPH_SCHEMA_SPEC.md §1, §2.8, §2.11 exemptions):

```sql
CREATE TABLE node_source_refs (
  node_id       uuid NOT NULL,        -- references a row in ANY node table (see note)
  node_type     text NOT NULL CONSTRAINT chk_node_source_refs_node_type
                  CHECK (node_type IN ('market','audience','problem','existing_solution','business_model','hypothesis')),
  evidence_id   uuid NOT NULL REFERENCES evidence(id),
  PRIMARY KEY (node_id, evidence_id)
);
CREATE INDEX idx_node_source_refs_evidence ON node_source_refs(evidence_id);
```

The `node_type` `CHECK` doesn't make `node_id` a real foreign key — it
only guards against a typo'd or garbage discriminator value (e.g.
`'banana'`), catching a real class of bug at the DB level cheaply. It
does not verify that `node_id` actually exists in the table
`node_type` names — that part is still application-enforced, as noted
below.

`node_id` cannot be a real foreign key (Postgres doesn't support FKs
that target "one of several tables"), so referential integrity here is
enforced at the application layer (the Orchestrator's schema
validation step, GRAPH_SCHEMA_SPEC.md §5) rather than the DB. This is
the one place DB constraints alone are insufficient — flagged so it
isn't assumed to be automatically enforced.

ALTER TABLE problem              ADD CONSTRAINT chk_status CHECK (status IN ('active','deprecated','archived','resurrection_candidate','merged'));
ALTER TABLE existing_solution    ADD CONSTRAINT chk_status CHECK (status IN ('active','deprecated','archived','resurrection_candidate','merged'));
ALTER TABLE business_model       ADD CONSTRAINT chk_status CHECK (status IN ('active','deprecated','archived','resurrection_candidate','merged'));
ALTER TABLE hypothesis           ADD CONSTRAINT chk_status CHECK (status IN ('active','deprecated','archived','resurrection_candidate','merged'));

## 6. Edges

### 6.1 Simple (binary) edges — one generic table

Most edge types in GRAPH_SCHEMA_SPEC.md §3 are genuinely binary
(`contains`, `has_audience`, `experiences`, `addressed_by`,
`competes_with`, `monetizes_via`, `supports`, `contradicts`,
`co_clustered_with`, `validates`, `invalidates`, `fits`,
`does_not_fit`, `promotes`, `supersedes`, `resolves`):

```sql
CREATE TABLE edge (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  edge_type      text NOT NULL,
  from_id        uuid NOT NULL,
  from_type      text NOT NULL,
  to_id          uuid NOT NULL,
  to_type        text NOT NULL,
  confidence     real,
  created_at     timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_edge_from ON edge(from_id, edge_type);
CREATE INDEX idx_edge_to   ON edge(to_id, edge_type);

-- `promotes` gets its own partial unique index, enforcing
-- "exactly one promotion per candidate" at the edge level too
-- (belt-and-braces alongside opportunity.promoted_from_candidate_id's
-- UNIQUE constraint in §3.10):
CREATE UNIQUE INDEX idx_promotes_once
  ON edge(from_id) WHERE edge_type = 'promotes';
```

`source_refs` on edges (required for most types per GRAPH_SCHEMA_SPEC.md
§3) reuse the same `node_source_refs` join table (§4) — an edge's `id`
is a valid `node_id` there, with `node_type = 'edge'`.

### 6.2 Hyperedges — dedicated junction tables, not the generic `edge` table

Two edge types are not truly binary and do not fit the `edge` table:

**`composes`** (5 fixed legs: Market, Audience, Problem, Hypothesis,
BusinessModel → one OpportunityCandidate):

```sql
CREATE TABLE opportunity_candidate_composition (
  candidate_id   uuid NOT NULL REFERENCES opportunity_candidate(id),
  node_id        uuid NOT NULL,
  node_type      text NOT NULL CHECK (node_type IN ('market','audience','problem','hypothesis','business_model')),
  role           text NOT NULL CHECK (role IN ('market','audience','problem','hypothesis','business_model')),
  PRIMARY KEY (candidate_id, role)
  -- PK on (candidate_id, role) rather than (candidate_id, node_id)
  -- enforces exactly one leg per role per candidate — a candidate
  -- cannot have two Market legs, directly encoding
  -- GRAPH_SCHEMA_SPEC.md §2.9's "exactly one of each" invariant.
);
```

**`reveals`** ((Problem, ExistingSolution set) → Hypothesis):

`existing_solution_id` is genuinely nullable — a hypothesis can cite
an absence of solution coverage, not just a specific competitor. A
nullable column cannot be part of a `PRIMARY KEY` in Postgres (PK
columns are implicitly `NOT NULL`); an earlier draft of this table
used `(hypothesis_id, problem_id, existing_solution_id)` as a
composite PK while claiming the last column was nullable — running it
confirmed Postgres silently coerces the column to `NOT NULL` the
moment it's part of a PK, quietly breaking the "cite an absence"
case. Fixed with a surrogate key and two partial unique indexes:

```sql
CREATE TABLE hypothesis_sources (
  id                     uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  hypothesis_id          uuid NOT NULL REFERENCES hypothesis(id),
  problem_id             uuid NOT NULL REFERENCES problem(id),
  existing_solution_id   uuid REFERENCES existing_solution(id)  -- nullable: absence-of-solution citation
);
CREATE UNIQUE INDEX idx_hypothesis_sources_with_solution
  ON hypothesis_sources(hypothesis_id, problem_id, existing_solution_id)
  WHERE existing_solution_id IS NOT NULL;
CREATE UNIQUE INDEX idx_hypothesis_sources_without_solution
  ON hypothesis_sources(hypothesis_id, problem_id)
  WHERE existing_solution_id IS NULL;
```

## 7. Enforcing Append-Only on Outcome (GRAPH_SCHEMA_SPEC.md §4.11)

Postgres has no native "insert-only" table mode. Enforce via a trigger
rather than relying on application discipline alone:

```sql
CREATE OR REPLACE FUNCTION forbid_outcome_mutation() RETURNS trigger AS $$
BEGIN
  RAISE EXCEPTION 'outcome rows are append-only — insert a new row, do not UPDATE or DELETE';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_outcome_no_update
  BEFORE UPDATE OR DELETE ON outcome
  FOR EACH ROW EXECUTE FUNCTION forbid_outcome_mutation();
```

This makes the immutability a DB-enforced guarantee, not just a
convention the application layer has to remember to respect — matches
the same "belt-and-braces" posture as the `promotes` unique index (§6.1).

## 8. The Promotion Transaction (Compression Agent's write path)

GRAPH_SCHEMA_SPEC.md §2.10 requires exactly one `Opportunity` row per
successful Compression run, created only via the `promotes` edge write
path. At the DB level this must be one transaction:

```sql
BEGIN;

UPDATE opportunity_candidate
  SET status = 'promoted'
  WHERE id = :winning_candidate_id AND status = 'candidate';
  -- the `AND status = 'candidate'` guard means this UPDATE affects
  -- zero rows (and the transaction should abort) if something else
  -- already promoted or deprecated this candidate — cheap optimistic
  -- concurrency check without a separate locking step

UPDATE opportunity_candidate
  SET status = 'deprecated'
  WHERE id = ANY(:losing_candidate_ids) AND status = 'candidate';

INSERT INTO opportunity (promoted_from_candidate_id, venture_score, confidence_score,
                          founder_fit_score, founder_fit_rationale, rationale_bullets, risk_summary)
  SELECT id, venture_score, confidence_score, founder_fit_score, founder_fit_rationale,
         :rationale_bullets, :risk_summary
  FROM opportunity_candidate WHERE id = :winning_candidate_id
  RETURNING id;

INSERT INTO edge (edge_type, from_id, from_type, to_id, to_type)
  VALUES ('promotes', :winning_candidate_id, 'opportunity_candidate', :new_opportunity_id, 'opportunity');

COMMIT;
```

If the app needs to check for an existing `active` Opportunity for an
overlapping lineage (to create a `supersedes` edge and flip it to
`superseded` — GRAPH_SCHEMA_SPEC.md §4.10), that check and the
resulting `UPDATE opportunity SET status = 'superseded'` +
`INSERT INTO edge (edge_type='supersedes', ...)` belong inside this
same transaction, not a follow-up one — otherwise a crash between the
two leaves two `active` Opportunity rows for the same lineage, which
GRAPH_SCHEMA_SPEC.md §4.10 says must never happen.

## 9. Indexing Priorities (beyond what's inline above)

- `evidence(cluster_id, cluster_version)` — composite index; every
  Validation Agent run groups by this pair (v3 §5.2).
- `evidence USING ivfflat (embedding vector_cosine_ops)` — **deferred**
  along with the `embedding` column itself (§3.1) until Phase 2 picks
  a specific NIM embedding model and its dimension. Not present in
  001_initial_schema.sql; comes in a follow-up migration.
- Partial indexes on `status = 'active'` for every structural node
  table (market, audience, problem, existing_solution,
  business_model, hypothesis) — the large majority of queries only
  care about the active subgraph; a full-table index would carry a
  lot of dead weight from `deprecated`/`archived` rows over time.
- `opportunity(status) WHERE status = 'active'` — should almost always
  return at most one row per lineage; if it's returning more, that's
  the bug described in GRAPH_SCHEMA_SPEC.md §4.10's invariant.

## 10. What's Deliberately Not Here

- **Migration tooling** — SQL migrations are the schema source of
  truth; Prisma is used for application access via introspection
  (`prisma db pull`/`generate`) only, never `prisma migrate` — raw SQL
  remains the escape hatch for anything Prisma can't express (advanced
  graph traversal, future vector similarity queries).
- **Read replicas / sharding** — premature at this stage; the graph
  size at MVP (v1 MVP.md scope: one vertical, 3 data sources) does not
  justify it, and adding it now is speculative infrastructure the
  freeze discipline from prior documents was specifically trying to
  avoid taking on early.
- **`evidence.embedding` column and its index** — deliberately deferred
  entirely (not just the model choice) until Phase 2 actually picks a
  specific NIM embedding model. Committing to a vector dimension (e.g.
  1536) before that choice is made would hard-code an assumption that
  might not match whatever model gets recorded in
  `model_routing_config` — different embedding models commonly use
  768, 1024, or 1536 dimensions, and changing a column's vector
  dimension later is a real migration, not a config change.

---

## 11. Summary

Postgres, table-per-node-type, universal UUID id space, one generic
binary-edge table plus two dedicated hyperedge junction tables for
`composes` and `reveals`, a polymorphic `node_source_refs` join table
for evidence citation, and two DB-enforced invariants (append-only
Outcome via trigger, exactly-one-promotion via unique index +
transaction) that turn GRAPH_SCHEMA_SPEC.md's structural rules into
things the database itself refuses to violate, not just things the
application is expected to remember.
